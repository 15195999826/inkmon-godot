根据文档和代码，让我解释这两个 agent 的使用场景：
---
🔥 Prometheus（普罗米修斯）- 战略规划器
是什么？
Prometheus 是一个规划专家，通过访谈模式与你交互，帮你制定详细的工作计划。
使用场景
| 场景 | 示例 |
|------|------|
| 大型功能开发 | "我要实现一个完整的用户认证系统" |
| 复杂重构 | "重构整个数据层，从 REST 迁移到 GraphQL" |
| 不确定如何开始 | "我想优化应用性能，但不知道从哪入手" |
| 需要分解任务 | "这个需求太大了，帮我拆分成可执行的步骤" |
如何使用
# 方式 1: 直接 @ 调用
@prometheus 帮我规划一个电商购物车功能的开发
# 方式 2: 使用 plan agent（如果 planner_enabled: true）
让 plan agent 帮我分析这个需求并制定计划
# 方式 3: 规划完成后启动工作
/start-work
工作流程
1. Prometheus 提问 → 了解你的需求、约束、偏好
2. 你回答 → 提供更多细节
3. 迭代几轮 → 直到需求清晰
4. 输出计划 → 生成结构化的工作计划（Markdown）
5. 执行计划 → 使用 /start-work 让 Sisyphus 执行
---
🗺️ Atlas（阿特拉斯）- 内部编排器
是什么？
Atlas 是内部编排逻辑（771 行代码），你通常不需要直接调用它。
它做什么？
Atlas 在后台自动处理：
- 任务分发和协调
- 后台 agent 管理
- 工具调用拦截和增强
- 上下文注入
- 错误恢复
你需要关心它吗？
不需要。Atlas 是 Sisyphus 的"大脑"，自动运行。
唯一可能需要配置的情况：
// 如果你想禁用 Atlas 编排逻辑（不推荐）
"disabled_hooks": ["atlas"]
---
📊 对比总结
| | Prometheus | Atlas |
|---|---|---|
| 角色 | 规划顾问 | 内部编排器 |
| 用户交互 | ✅ 直接对话 | ❌ 后台运行 |
| 调用方式 | @prometheus 或 plan | 自动 |
| 输出 | 工作计划（Markdown） | 无直接输出 |
| 使用场景 | 大型任务规划 | 所有任务的底层协调 |
---
🎯 实际使用建议
什么时候用 Prometheus？
✅ 用 Prometheus:
- "我要从零开始构建一个博客系统"
- "帮我规划这个季度的技术债务清理"
- "这个需求很复杂，先帮我理清思路"
❌ 不需要 Prometheus:
- "修复这个 bug"（直接让 Sisyphus 做）
- "添加一个按钮"（太简单）
- "ulw 实现登录功能"（ultrawork 会自动处理）
典型工作流
1. 你: @prometheus 我想实现一个实时聊天功能
2. Prometheus: 我来帮你规划。先问几个问题：
   - 需要支持多少并发用户？
   - 是否需要消息持久化？
   - 使用什么技术栈？
   ...
3. 你: 回答问题...
4. Prometheus: 输出详细计划：
   ## 实时聊天功能开发计划
   ### Phase 1: 基础架构
   - [ ] 设置 WebSocket 服务器
   - [ ] 实现连接管理
   ...
5. 你: /start-work
6. Sisyphus: 开始按计划执行...
---
简单记忆：
- Prometheus = 规划师，帮你想清楚要做什么
- Atlas = 幕后英雄，你不用管它
